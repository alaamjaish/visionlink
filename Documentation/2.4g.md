## TL;DR (read this first)

By **renaming Alaa's iPhone to broadcast the same SSID + password as his home Wi-Fi**, the Raspberry Pi automatically joined the iPhone hotspot thinking it was home. SSH worked end-to-end after fixing two cached references on Alaa's laptop (an old IP in SSH config; stale host keys in `known_hosts`).

## What we actually did, in execution order

This is the literal sequence of moves we took today, with their outcomes. Use this as the playbook to repro elsewhere.

### Phase 1 — Diagnose why the Pi was unreachable

1. **Started by trying Ethernet + Windows ICS** (Method 2 from the original options). Wired the Pi to laptop via Ethernet. Enabled Windows Internet Connection Sharing.
2. **Discovered the Ethernet network was set to "Public"** profile on Windows → firewall blocking ICS DHCP responses. Set Ethernet adapter to "Private" via an elevated PowerShell. Restarted SharedAccess service.
3. **Pi was actively broadcasting DHCP DISCOVERs** (visible in adapter counters: `ReceivedBroadcastPackets` climbed from 0 → ~100). **Windows was replying with DHCP OFFERs** (visible as `SentBroadcastPackets` going from 0 → ~1800). **But the Pi never accepted any offer** — `ReceivedUnicastPackets` stayed at 0. No lease ever completed.
4. **Tried switching laptop Wi-Fi from school to phone hotspot** to see if school network was somehow interfering. No change — same failure pattern.
5. **Conclusion:** the Pi's `eth0` interface rejects DHCP offers from Windows ICS (probably because there's no NetworkManager wired profile configured to accept it, or it expects a specific gateway IP). This is a Pi-side issue that cannot be fixed without physical access to the Pi. **Ethernet was a dead end.**

### Phase 2 — Pivot to the SSID-spoofing trick

6. **Renamed Alaa's iPhone** via `Settings → General → About → Name` to exactly: `Turksat Kablonet R9HF 5G` (case-sensitive, including the "5G" suffix, including the spaces).
7. **Changed iPhone hotspot password** via `Settings → Personal Hotspot → Wi-Fi Password` to exactly match Alaa's home Wi-Fi password.
8. **Ensured "Maximize Compatibility" was ON** in `Settings → Personal Hotspot`. This forces the hotspot to **2.4 GHz**, which the Pi's wlan0 supports cleanly. (Hence the nickname for this whole approach: "2.4G".)
9. **Ensured "Allow Others to Join" was ON** in the hotspot settings.
10. **Confirmed the laptop was connected to the renamed hotspot** (so we'd be on the same network as the Pi once it joined).
11. **Power-cycled the Pi** — unplugged USB-C, waited 5 seconds, plugged back in. Pi booted, scanned available Wi-Fi networks, saw the spoofed SSID, matched it against its saved profile (home Wi-Fi), and **joined automatically.**
12. **Verified Pi was connected** by checking the iPhone's Personal Hotspot screen → showed "2 devices connected" (laptop + Pi).

### Phase 3 — Fix the laptop-side SSH path

When VS Code Remote-SSH tried to connect, the first error was a **connection timeout** because the SSH config had the Pi's old home IP hardcoded:

```
ssh: connect to host 192.168.0.31 port 22: Connection timed out
```

13. **Found the Pi's current IP on the hotspot** by running `arp -a` from PowerShell. The iPhone hotspot uses the `172.20.10.0/28` subnet:
    - Laptop: `172.20.10.3`
    - Pi: `172.20.10.5`
14. **Confirmed mDNS works** over the iPhone hotspot — `ping visionlink.local` resolved to `172.20.10.5` with ~9 ms RTT.
15. **Edited `~/.ssh/config`** (at `C:\Users\Alaa M. Jaish\.ssh\config`) — changed `HostName 192.168.0.31` to `HostName visionlink.local`. This makes SSH portable across networks via mDNS resolution.

### Phase 4 — Resolve the host key mismatch

Next SSH attempt: KEX succeeded, host key received, but `known_hosts` had a stale ECDSA entry on line 4 for `visionlink.local` from a previous Pi or reflash. SSH refused to continue with:

```
WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!
Host key for visionlink.local has changed and you have requested strict checking.
```

16. **Inspected `known_hosts`** — found six stale entries total: three under `visionlink.local` (ED25519, RSA, ECDSA) plus three under `192.168.0.31` (same three algorithms).
17. **Removed the stale entries cleanly** via `ssh-keygen -R`:
    ```powershell
    ssh-keygen -R visionlink.local
    ssh-keygen -R 192.168.0.31
    ```
    Both backed up to `known_hosts.old`. File went from 11 lines to 5 lines (legit entries preserved: github.com, etc.).

### Phase 5 — Pre-populate the new host key (avoid interactive prompt in VS Code Remote-SSH)

18. **Attempted `ssh-keyscan visionlink.local`** to fetch the Pi's current host keys non-interactively. **It failed** with `choose_kex: unsupported KEX method sntrup761x25519-sha512@openssh.com`. The Pi runs `OpenSSH 10.0p2` which prefers post-quantum KEX; Windows OpenSSH 9.5's `ssh-keyscan` can't negotiate that.
19. **Worked around it** by using regular `ssh` with `StrictHostKeyChecking=accept-new` and auth disabled — `ssh` (unlike `ssh-keyscan`) successfully negotiates an alternate KEX, then auto-accepts and caches the host key, then fails at auth (as designed):
    ```powershell
    ssh -o "StrictHostKeyChecking=accept-new" -o "BatchMode=yes" -o "PasswordAuthentication=no" -o "PubkeyAuthentication=no" visionlink "echo CONNECTED"
    ```
    Output: `Warning: Permanently added 'visionlink.local' (ED25519) to the list of known hosts.` followed by `Permission denied (publickey,password).` — host key cached, auth refused as intended.
20. **Confirmed the new key matches what was previously cached under `192.168.0.31`** (byte-identical ED25519 key in `known_hosts`). This proves the Pi was the same physical machine the whole time; the stale visionlink.local entries were from an earlier setup or reflash.

### Phase 6 — SSH works

21. Alaa ran `ssh visionlink` from PowerShell. Got the password prompt. Typed his Pi password. **SSH session established.** He's now logged into the Pi over the iPhone hotspot.

**End-to-end time from "Pi unreachable" to "Pi SSH'd in":** roughly two hours, most of it spent ruling out Ethernet before pivoting to Wi-Fi spoofing.

## The team resilience plan ("visionlink" SSID convention)

This is the strategic plan for demo day so we never get blindsided by a single phone or single SSID failing.

### The convention

Every team phone (Alaa, Ali Salih, F-Alaa, Defne, and a fallback family/spare phone) renames its hotspot to:

- **SSID:** `visionlink` (all lowercase, no spaces, no special chars)
- **Password:** `visionlink` (all lowercase, no spaces)
- **Security:** WPA2-Personal (the default on iPhone / Android hotspots)
- **Band:** 2.4 GHz (toggle "Maximize Compatibility" ON on iPhone)

This SSID/password is **demo-only** — not for any sensitive use. The password is intentionally short and memorable so anyone on the team can spin up a hotspot in seconds.

### What this gives us

Once the Pi has `visionlink` saved as a Wi-Fi profile (next section), then on demo day:

- Whichever team member's phone is closest to the Pi → they turn on their hotspot named `visionlink` → Pi auto-joins.
- If that phone dies / runs out of data / overheats → the next team member's phone takes over with the same SSID → Pi seamlessly reconnects.
- If a venue has unstable Wi-Fi → still doesn't matter. Phone hotspot wins.

The Pi will know **at least two networks**: home Wi-Fi (`Turksat Kablonet R9HF 5G`) AND `visionlink`. Anywhere either of those is in range, the Pi connects automatically.

### How to add the `visionlink` profile to the Pi

While SSH'd in (right now), run on the Pi:

```bash
# Add a new WPA2-PSK profile without trying to connect yet
sudo nmcli connection add type wifi con-name "visionlink" ifname wlan0 ssid "visionlink"
sudo nmcli connection modify "visionlink" wifi-sec.key-mgmt wpa-psk
sudo nmcli connection modify "visionlink" wifi-sec.psk "visionlink"
sudo nmcli connection modify "visionlink" connection.autoconnect yes
sudo nmcli connection modify "visionlink" connection.autoconnect-priority 50

# Verify it's saved
nmcli -f NAME,TYPE,AUTOCONNECT,AUTOCONNECT-PRIORITY connection show
```

You should see both `Turksat Kablonet R9HF 5G` and `visionlink` listed with `AUTOCONNECT yes`.

(The home Wi-Fi has higher implicit priority because it was used most recently. If you want to bias the Pi toward home when both are in range, also set: `sudo nmcli connection modify "Turksat Kablonet R9HF 5G" connection.autoconnect-priority 100`.)

### Test the failover before demo day

Don't skip this. Demo failures from untested failover paths are how teams lose presentations:

1. Pi is currently on the spoofed `Turksat Kablonet R9HF 5G` (your iPhone hotspot).
2. Rename your iPhone back to its real name (e.g., `iPhone 2`) so it stops broadcasting the spoofed SSID. Hotspot goes away.
3. Now broadcast a new hotspot from the same iPhone (or any team phone) with SSID `visionlink` and password `visionlink`.
4. Wait ~30 seconds. The Pi should auto-join the new `visionlink` hotspot.
5. From your laptop (also connected to the same `visionlink` hotspot), run `ssh visionlink`. Should work identically.
6. Repeat with each teammate's phone — confirm the Pi joins each in turn when others are powered down.

If any test fails, fix it now, not on demo day.

## Gotchas encountered today (read these before reproducing this elsewhere)

These are non-obvious things that bit us. Future-Alaa or future-agent: save yourself the time.

1. **iPhone hotspot SSID is *not* set in the Personal Hotspot menu.** It's the iPhone's *device name* (Settings → General → About → Name). Most people look for it in the wrong place first.
2. **"Maximize Compatibility" must be ON** for the iPhone to broadcast 2.4 GHz. Without it, iPhone may default to 5 GHz, which works on Pi 4B *most of the time* but has edge cases on certain Wi-Fi adapter firmware.
3. **iPhone hotspot uses a fixed subnet: `172.20.10.0/28`.** That's only 14 usable IPs — you can have at most ~6 devices on it. Plenty for a Pi + laptop + a phone. The Pi gets some address in `172.20.10.2` through `172.20.10.14`, usually `.5`.
4. **mDNS over iPhone hotspot works** on current iOS. Older versions used to block multicast between hotspot clients. If you're on iOS 14 or older, this might not work and you'd need to use the IP directly.
5. **Windows ICS does NOT pass IPv6.** Don't bother trying `ping -6 ff02::1` over an ICS-shared adapter; it fails with "Transmit failed. General failure." (We learned this the hard way.)
6. **Windows OpenSSH 9.5's `ssh-keyscan` can't negotiate post-quantum KEX** that the Pi's OpenSSH 10.0p2 prefers. Use `ssh -o StrictHostKeyChecking=accept-new` instead (the regular `ssh` is more flexible about KEX algorithms).
7. **Old `known_hosts` entries persist forever.** If you ever reflash the Pi or change anything that regenerates its SSH host keys, you'll hit "REMOTE HOST IDENTIFICATION HAS CHANGED" — clean with `ssh-keygen -R visionlink.local`.
8. **VS Code Remote-SSH is non-interactive.** It can't show password prompts or fingerprint-accept prompts mid-handshake. Always sanity-check SSH from a regular PowerShell terminal first, accept any prompts there, then connect Remote-SSH.
