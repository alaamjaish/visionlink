## 1. Voice agent reliability — the big problem we solved

### The lie
Earlier in the day, we'd ask Gemini to *"log an incident — broken phone"* and it would respond *"Got it, I've logged the broken phone as an equipment incident."* Sounded perfect. **Nothing was actually written to the database.** The model was generating polite "I did it" language without firing the `log_incident` tool. Same problem on the very first agent test (it said pump A3 torque was 42 Nm — actually 28 Nm — purely hallucinated).

Root cause: `gemini-3.1-flash-live-preview` in audio mode prioritises conversational fluency over tool calling. ComplexFuncBench Audio scores 90.8% on multi-step calling, but our case was hitting the 9.2% failure tail because the system prompt allowed wiggle room.

### The fix (three-pronged)

**(a) Rewrote the system prompt with three "ABSOLUTE RULES" at the top:**
1. **Never lie about completing actions** — if you say "I logged it" without a tool call, that's the worst possible failure. If you can't fire the tool, say so explicitly.
2. **Verb triggers** — explicit list of words (`log`, `mark`, `request`, `order`, `email`, `send`, `tell me about`) that *force* a tool call regardless of topic. Even non-factory items like "I broke my phone" must trigger `log_incident`.
3. **Info-collection trap** — after asking a clarifying question and getting an answer, the *very next action* must be the tool call. No more "Got it" without then calling.

Default prompt is now **5,201 chars** (was ~700 chars before today). Lives at `dashboard/server.py:DEFAULT_SYSTEM_PROMPT`.

**(b) Added a glaring visible badge to the dashboard transcript:**
Every time a tool actually fires, a green left-bordered card slides into the transcript with the exact `tool_name(args)`. If you don't see the badge, the agent didn't call the tool — full stop. CSS class `.msg.tool` in `dashboard/static/index.html`.

**(c) Diagnostic logging in the receive loop:**
- Every Live API message dumps its populated attribute names, so when something goes wrong we can see *what Gemini actually sent*.
- Mic blocks log per-second peak (`raw_peak` and `clean_peak`) so we know the mic is delivering audible voice vs. silence.

### Things that DID NOT work (tried and reverted)
- **`tool_config=ToolConfig(function_calling_config=...)`** on `LiveConnectConfig` — pydantic rejected it as `extra_forbidden`. Live API in google-genai 1.73 doesn't accept this field. Removed.
- **Switching to `gemini-2.0-flash-live-001` / other model names** — none exist for v1alpha bidiGenerateContent. We're stuck with `gemini-3.1-flash-live-preview`.

### Result
Tool fires reliably for verb-triggered phrases. Hallucination rate ≈ 0 in casual tests. The honesty fallback ("Sorry, I couldn't log that — let me try again") fires when something downstream actually breaks, instead of fake confirmation.

### Critical bug fixed: import-order

`tools.py` originally read `SMTP_EMAIL` and `SMTP_APP_PASSWORD` at module-level (`os.getenv(...)` constants). Problem: `dashboard/server.py` imports `tools.py` *before* calling `load_dotenv()`. So tools.py would cache empty strings, then the handler would error with "no email provider configured" even when `.env` had everything.

**Fix:** moved env reads into a `_email_config()` function called inside the handler. Now reads are always fresh.

## 5. Hardware audio fixes (the painful detective work)

### The ghost: 2070

Mic was delivering a steady DC-offset of ~2070 (≈6% full scale) regardless of whether the user was speaking or silent. Voice modulation was buried below the noise floor. Gemini's VAD was either failing to commit turns (30+ second hangs) or transcribing as silence.

This was a solved problem in the SPH0645LM4H community for years — the mic outputs 24-bit data MSB-justified in a 32-bit slot, and the ALSA plug down-cast to int16 doesn't remove the DC bias. We just hadn't hit it.

**Fix in `dashboard/server.py:send_mic`:**
```python
arr = np.frombuffer(raw, dtype=np.int16).astype(np.int32)
arr = arr - int(arr.mean())                # DC removal
arr = np.clip(arr * MIC_GAIN, -32768, 32767).astype(np.int16)  # 6x gain
```

`MIC_GAIN` defaults to `6.0` (configurable from the Agent Settings panel). Voice now reaches ~50% full scale; silence drops to <1%. Gemini hears clearly. VAD fires in <500ms.

### Speaker output

Gemini Live TTS output was quiet on the MAX98357A → 8Ω speaker chain. Added software gain in `dashboard/audio_worker.py`:
- `SPEAKER_GAIN = 5.0` (boosts ~14 dB)
- Soft clip at 96% full scale
- Stereo expansion (some I2S amps only listen to one slot)

`SPEAKER_GAIN` is a constant in the worker subprocess, not yet live-tunable from the UI (worker is a separate process, would need IPC). Live tunable later.

### `~/.asoundrc` mystery

Mid-debug, we hit `[audio_worker] primary device open failed: 'mic_left' not found in PortAudio enumeration`. The asoundrc had vanished. Was definitely there earlier in the session (PortAudio resolved `mic_left=5` when voice was working). Possibly cleared by VS Code or a stray reboot.

Re-wrote it from the AUDIO_DEBUG_HANDOFF doc spec:
```
pcm.!default { type asym; playback.pcm "speaker"; capture.pcm "mic_left" }
pcm.speaker  { type plug; slave.pcm "hw:3,0" }
pcm.mic_left { type plug; slave.pcm "hw:3,0" }
pcm.mic_right { ... }
pcm.mic_left_raw { type hw; card 3; device 0 }
ctl.!default { type hw; card 3 }
```

Lives at `/home/visionlink/.asoundrc`. **NOT in the repo** (it's user/system config).

## 13. Lessons / non-obvious knowledge for next time

1. **`gemini-3.1-flash-live-preview` lies in audio mode unless the prompt is brutal.** "Don't lie about completing actions" is the load-bearing instruction. Soft prompts produce hallucinations.

2. **The SPH0645 has a constant DC offset (~2070 in int16)** that buries voice signal under the ALSA plug's int16 cast. Always subtract block mean before sending audio anywhere VAD-aware.

3. **`tool_config` is for non-Live API only** in google-genai 1.73. `LiveConnectConfig` rejects it with `extra_forbidden`. Drive tool behaviour from the system prompt instead.

4. **Always restart the dashboard after editing `.env`.** `os.environ` is cached at module load; `load_dotenv` only writes once at startup. Re-saving `.env` doesn't touch the running process.

5. **Tool calls in Gemini Live arrive on `message.tool_call`, NOT in `server_content`.** The receive loop must check this attribute *before* the `if sc is None: continue` short-circuit — otherwise tool messages get silently dropped.

6. **Supabase RLS defaults to ON for new tables.** Anon-key clients (the browser ops dashboard) will see zero rows unless RLS is disabled or policies allow. Service key bypasses RLS, which masked this bug for hours (Pi could read everything; browser saw "Loading…").

7. **The Supabase SQL editor's dollar-quote handling is unreliable** for big HTML strings with nested `'` and `"`. Use a Python seeder via the supabase-py client instead — bypasses all editor escaping.

8. **`useRealtimeRows` works best when the dependency array is fully primitive** (string table name, boolean ascending, number limit). Pass an options object → re-renders → effect re-runs → channel thrash.

9. **Next.js 16 dev server blocks cross-origin HMR by default.** Add `allowedDevOrigins: ['<lan-ip>']` to `next.config.ts` so a browser on a different LAN device can hot-reload.

10. **Pydantic FunctionDeclaration types want UPPERCASE Schema enums** (`Type.OBJECT`, `Type.STRING`). Lowercase ("object", "string") works for the dict form but breaks the typed builder. Our `_TYPE_MAP` in `tools.py` handles this.
