The Six Buttons

Two modes, six handlers, three gestures (single, double, hold). Each button is a falling-edge GPIO interrupt that turns into an async Python coroutine.

B1 · GPIO 23 · Pin 16

Documentation Session

Opens or closes a shift session. Everything captured while it's open (photos, videos, voice notes) is grouped under the same session_id for the supervisor.

Press journey

  1. Worker presses B1. The GPIO pin drops from 3.3 V to 0 V.
  2. The Linux kernel raises a falling-edge interrupt on GPIO 23.
  3. scripts/gpio_bridge.py debounces (~20 ms LOW + 40 ms HIGH) and POSTs /api/button/1/single to the FastAPI dashboard.
  4. The handler b1_doc_session_toggle reads current state. No session open? Insert a new sessions row. Session open? Update status='closed', ended_at = now().
  5. Supabase's logical replication pushes the change over Realtime. The supervisor's dashboard reflects the new state within ~200 ms.
Guard rail: An 800 ms cooldown stops mechanical bounce from being misclassified as a deliberate second press that would close the session right after opening it.
B2 · GPIO 25 · Pin 22

Photo / Video

A single press is a JPEG. A double press is a 5-second MP4. The double-press window is 700 ms.

Photo journey

  1. B2 single press → b2_take_photo
  2. picamera2 captures a 1280×720 JPEG at quality 85
  3. Bytes upload to session-assets/<session_id>/photo_<ts>.jpg
  4. session_assets row inserted (kind='photo')
  5. Supervisor sees the thumbnail in ~3 s end-to-end

Video journey

  1. B2 double press → b2_record_video
  2. rpicam-vid --codec libav --libav-format mp4 -t 5000 writes a 5 s MP4 (1280×720 @ 30 fps)
  3. MP4 uploads to the same bucket
  4. session_assets row with kind='video', duration_s=5
  5. Hard cap at B2_VIDEO_MAX_DURATION (default 300 s) prevents runaway captures
Why no audio in the video. The I²S microphone is held exclusively by the always-running audio_worker subprocess that feeds AI sessions and B3 voice notes. A parallel arecord would fail with "device busy". An audio mux through the existing AudioBridge is a planned improvement.
B3 · GPIO 22 · Pin 15

Voice Note

Press to start, press again to stop. PCM 16 kHz mono from the SPH0645 I²S MEMS microphone, wrapped in a WAV header on upload.

  1. B3 single press → b3_voice_note_start
  2. The handler subscribes to the AudioBridge mic stream and starts buffering raw PCM in memory
  3. Worker speaks naturally (e.g. "check valve B7 tomorrow morning")
  4. B3 pressed again → b3_voice_note_end
  5. Buffered PCM is wrapped in a WAV header (16 kHz, 16-bit, mono) and uploaded to session-assets/<session_id>/voice_<ts>.wav
  6. session_assets row inserted with kind='voice_note' and computed duration_s
  7. Supervisor sees an inline playable <audio> element on the dashboard
Microphone exclusivity: B3 refuses to start while any AI session (B4, B5, or B6) is using the mic. The worker gets an explicit error rather than a corrupted recording split between two consumers.
B4 · GPIO 5 · Pin 29

AI Voice Agent

Bidirectional streaming voice with a cloud AI. Default is Gemini Live; can be switched to OpenAI Realtime from the supervisor's Wearable Settings panel.

  1. B4 single press → b4_ai_voice_only
  2. Handler reads wearable_settings.b4_provider from Supabase — gemini or openai
  3. WebSocket opens to gemini-3.1-flash-live-preview (Gemini Live API) or gpt-realtime-2 (OpenAI Realtime API)
  4. AudioBridge forwards mic audio → model. Model responses → speaker. Mic is muted while TTS plays (half-duplex echo suppression — no AEC).
  5. All six function-calling tools exposed: lookup_component, log_incident, mark_task_complete, get_my_assignments, request_part, send_report
  6. Live transcripts stream over WebSocket to the voice command center and appear in real time
  7. B4 double-press → close WebSocket, stop audio routing
Three absolute rules in the system prompt stop the most common hallucinations: (1) Never lie about completing actions — saying "I logged it" requires calling the matching tool first. (2) Verb triggers — words like log, mark, send, request fire the matching tool immediately. (3) One clarifying question max — commit with reasonable defaults instead of stalling.
B5 · GPIO 6 · Pin 31

AI Voice + Vision

The most powerful button. Same as B4 but the AI session is configured with a vision channel — the model sees what the camera sees.

  1. B5 single press → b5_ai_voice_vision
  2. WebSocket opens to the provider with vision enabled
  3. Default mode is snap-on-press: the first B5 press starts the session and triggers an automatic JPEG snapshot; subsequent single presses inject fresh frames into the same conversation
  4. Alternative modes (from Wearable Settings): gemini_video (continuous 1 fps stream, Gemini-only) and auto_snap_4s (auto-snap every 4 s, both providers)
  5. Worker asks things like "what's wrong with this gauge?" or "what's this part code?"
  6. JPEGs travel over the same WebSocket as audio — for OpenAI, as base64 input_image; for Gemini, as Part.from_bytes(mime_type="image/jpeg")
B6 · GPIO 13 · Pin 33 · SOS

SOS Panic Mode

B6 has two behaviors. A single press is a pocket-dial warning toast. A double press is the most complex flow in the system.

SOS double-press journey

  1. B6 double press → b6_sos_trigger
  2. A row is inserted into sos_events with triggered_at = now(). The row's UUID becomes the SOS ID.
  3. Email the safety officer. A fire-and-forget task uses the send_report tool internally — looks up sos_alert_recipient_role (default "safety officer") in the managers table and sends via Gmail SMTP.
  4. Auto-close any open documentation session. Worker shouldn't be doing paperwork in an emergency, so any open B1 session is updated to status='closed' automatically.
  5. Start an AI emergency session. Gemini by default (configurable via sos_provider); falls back to OpenAI gpt-realtime-2 if Gemini deps are missing. OpenAI path uses the warm "marin" voice.
  6. Inject an emergency-specific system prompt. The model opens the call first: "Alaa, I'm here with you. Help is on the way. Are you hurt? Where are you?" — 1–2 short sentences per turn, calm tone, no urgency in the voice.
  7. Auto-snap a camera frame every sos_photo_interval_s seconds (default 10 s for Gemini, 4 s for OpenAI). Each frame uploads to sos/<sos_id>/frame_<ts>.jpg and is injected into the AI session via send_image(). The frames_sent counter increments.
  8. Stream the live transcript of what the AI says into sos_events.live_transcript. The supervisor watches the stream appear character-by-character.
  9. The supervisor's SOS panel goes red and pulses with a Web Audio API two-tone alarm.
  10. Supervisor shuts it off. A red SHUT OFF button opens a modal asking for the supervisor's name and reason. On confirm, resolved=true. The wearable polls every 2 s — within ~2 s, the AI session closes, the auto-snap loop stops, and the wearable falls silent.
  11. Hard timeout. If the supervisor never sees the alert, asyncio.wait_for auto-resolves after sos_max_duration_s (default 600 s = 10 minutes) to bound audio session length and storage cost.
Pocket-dial protection. A single press is intentionally a no-op (with a toast warning) so that brushing the wearable against a wall doesn't trigger an emergency. A real SOS requires the deliberate two-tap pattern.
